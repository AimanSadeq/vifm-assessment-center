# VIFM-AC Job-Analysis Kit (Phase 1)

**Purpose:** Derive the v2 competency framework from VIFM's own role evidence —
not from any third-party model. This kit is what turns the v2 draft
(`docs/competency-framework-v2-design.md`) from a *re-worded scaffold* into a
framework VIFM can defend as **original** and **job-related**.

> **Why this phase is non-negotiable.** Re-wording the old competencies does not
> make the framework original or valid. Originality and content validity are
> established here, by collecting primary evidence and **allowing it to change
> the competency set** — merging, splitting, adding, and dropping competencies
> based on what the roles actually require.

---

## 1. What this kit produces

| Output | Feeds |
|---|---|
| Role context + accountabilities per role family | Role profiles; assessment design |
| Critical incidents (effective / ineffective behaviours) | Behavioural indicators; BARS anchoring (Phase 2) |
| Card-sort results (criticality + structural edits) | The **final competency set** (resize decisions) |
| Competency ratings (importance / frequency / essential) | Content-validity evidence (Phase 3, CVR) |
| Completed role profiles | Client engagements; tiered profiles (Blueprint §3.3) |

## 2. Method (multi-method, by design)

Single-source job analysis is weak. We triangulate four instruments:

1. **SME structured interview** (`01-sme-interview-guide.md`) — role context and
   what differentiates strong performers.
2. **Critical Incident Technique** (`02-critical-incident-template.md`) — concrete
   behaviours, the raw material for indicators and BARS.
3. **Competency card-sort** (`03-competency-card-sort.md`) — criticality *and*
   the structural edits that resize the framework.
4. **Competency rating form** (`04-competency-rating-form.md`) — quantitative
   relevance + Lawshe essential ratings for CVR.

Outputs are captured in the **role profile template**
(`05-role-profile-template.md`).

## 3. Sampling plan

- **Role families first, not every job title.** Start with the target GCC
  banking/finance role families (e.g., Relationship/Commercial banking,
  Risk & Compliance, Operations, Finance/Treasury, Branch/Retail leadership,
  Credit). Confirm the list with the client/stakeholders.
- **SMEs per role family:** aim for **6–10** — a mix of **high-performing
  incumbents** and **their supervisors** (supervisors see the
  performance differentiators; incumbents see the task reality).
- **Minimum for content validity (Phase 3 CVR):** Lawshe tables require a
  defined panel size; **≥ 8 raters** is a practical floor (see
  `04-competency-rating-form.md` for critical values).
- **Guard against bias:** mix tenure, gender, and business unit; avoid drawing
  all SMEs from one team or one manager's reports.

## 4. Workflow

```
For each role family:
  1. Brief + consent (script in 01)
  2. SME interview (45–60 min)        → role context, accountabilities, differentiators
  3. Critical incidents (within/after interview) → 6–10 incidents per SME
  4. Card-sort (15–20 min)            → criticality bands + structural edits
  5. Rating form (10–15 min)          → importance/frequency/essential
  6. Capture into role profile (05)
After all SMEs:
  7. Aggregate + apply decision rules (§5)
  8. Update v2 framework set + behavioural indicators
```

## 5. Decision rules for the final competency set

Apply *after* aggregating across SMEs and instruments. A competency is
**retained for a role** when it meets the threshold on convergent evidence:

- **Include** if: mean Importance ≥ 4.0 (of 5) **and** card-sort criticality is
  "Essential/Important" for a majority of SMEs **and** it is supported by ≥ 1
  critical incident.
- **Merge** two competencies if SMEs repeatedly rate/sort them as the same thing
  or incidents don't distinguish them.
- **Split** a competency if incidents reveal two clearly distinct behaviour sets
  scored differently.
- **Add** a competency if incidents/interviews surface a required capability not
  in the draft (likely for GCC banking: e.g., regulatory/Sharia awareness, AML
  vigilance, client-confidentiality discipline).
- **Drop** if Importance < 3.0 and rarely sorted "Essential."

> **Expect — and document — at least some deviation from the 38-item parity.**
> A final set identical to the prior 38 is a red flag for "this is just a
> rename." Record every merge/split/add/drop with its evidence in the analysis
> log; that log is your originality and validity audit trail.

## 6. Standards & ethics

- **Standards:** AERA/APA/NCME *Standards*; ISO 10667 (assessment process); ITC
  Guidelines (Test Use; Test Adaptation for the Arabic versions).
- **Informed consent:** every SME consents to participation and recording; data
  used only for framework development.
- **Confidentiality & data protection:** handle SME and incident data per UAE
  Federal Decree-Law No. 45/2021 and KSA PDPL (as the Blueprint commits). Anonymise
  incidents before they enter indicator libraries.
